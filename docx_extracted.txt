Section 1: What Are We Building and Why
1.1 The Problem We Are Solving
Algoleap has around 30 to 50 important files — pitch decks, demo videos, case studies, architecture documents, and templates. These files currently live in OneDrive (Microsoft's cloud storage), but there is no organised way for employees to find them. If a sales person needs a case study about supply chain AI, they have to ask a colleague or search through folders manually. This wastes time.
1.2 The Solution We Are Building
We are building an internal website (a web portal) that acts like a smart, searchable 
catalog
 of all those files. Think of it like a library index card system — except it is on the web, it has a search bar, and it shows you exactly where to find each document.
CONCEPT  What This Means
A web portal is just a website that is private — only certain people (in this case, Algoleap employees) can log in and use it.
The portal does NOT store the actual files. It only stores information ABOUT each file (title, description, type, etc.) and a link to where the file lives in OneDrive.
Think of it as a menu at a restaurant. The menu does not contain the food — it just tells you what is available and how to get it.
1.3 Who Will Use This Portal
User
What They Need the Portal For
Sales & BD Team
Finding pitch decks and case studies to use in client meetings
Delivery / Engineering Teams
Finding architecture docs, templates, and reusable technical assets
Leadership
Finding company credentials, capability decks, and engagement summaries
HR & People Team
Finding onboarding materials, training docs, and process guides
Knowledge Management Owner
Adding new artifacts, editing existing ones, archiving old ones
1.4 What the Portal Will Look Like (Features)
Here is a plain-English list of every feature we will build:
A login page — employees sign in with their Algoleap Google account
A sidebar — with filters to narrow down artifacts by type, service line, audience, and status
A search bar — type any keyword to instantly find matching artifacts
Artifact cards — each file is shown as a card with its title, description, type badge, tags, and an 'Open' button that opens the OneDrive link
A stats bar — shows total artifacts, how many are current, how many need review
Grid and list view toggle — switch between card grid layout and a compact list layout
An artifact detail panel — click any card to see its full metadata in a slide-out panel
An admin panel — for the knowledge management owner to add, edit, and archive artifacts
1.5 What We Are NOT Building (Important Boundaries)
CAUTION  Watch Out
We are NOT building a file storage system. Files stay in OneDrive. We only store links.
We are NOT building a document editor. Employees open files in OneDrive to edit them.
We are NOT building an AI chatbot in this phase. That is planned for a future phase.
We are NOT building a public-facing website. Only @algoleap.com email accounts can log in.
Section 2: Your Tools — What Each One Does
Before you write a single line of code, you need to understand every tool in your toolkit. This section explains each tool in plain English. Do not skip this — understanding WHY you are using a tool will help you debug problems faster.
2.1 Next.js — The Main Framework
CONCEPT  What This Means
Next.js is a framework built on top of React (a popular JavaScript library for building user interfaces).
Think of React as the engine of a car, and Next.js as the full car — it adds routing (how pages are linked), server-side rendering (preparing pages on the server before sending to the browser), and many other helpful features.
We use Next.js because it works perfectly with 
Vercel
 (where we deploy) and handles many complex things automatically.
What you will use it for: Building every page of the portal — the login page, the 
catalog
 page, the admin page, and all the UI components (cards, sidebar, search bar, etc.).
2.2 Tailwind CSS — Styling the UI
CONCEPT  What This Means
CSS is the language that controls how a website looks (
colors
, fonts, spacing, layout).
Tailwind CSS is a toolkit of pre-written CSS classes. Instead of writing custom CSS, you add short class names directly to your HTML elements.
For example: <div class="bg-green-900 text-white p-4"> applies a dark green background, white text, and padding — all without writing a single line of custom CSS.
This makes styling fast and consistent across the entire portal.
What you will use it for: All visual styling — 
colors
, fonts, spacing, card layouts, responsive design (mobile vs desktop).
2.3 
Supabase
 — Your Database and Login System
CONCEPT  What This Means
A database is a place to store and retrieve structured data. In our case, we will store the details of every artifact (title, description, OneDrive link, tags, etc.).
Supabase
 gives us a PostgreSQL database (a very popular and reliable type of database), a built-in authentication system (handles logins securely), and a JavaScript SDK (a set of pre-written functions to interact with the database from your code).
Think of 
Supabase
 as a combination of a filing cabinet (database) and a security guard (authentication) — all provided as a free cloud service.
You do not need to set up your own server. 
Supabase
 handles all of that.
What you will use it for: Storing all artifact records, managing user logins, and enforcing security rules (e.g., only admins can add new artifacts).
2.4 Google OAuth — The Login Method
CONCEPT  What This Means
OAuth is a secure way to let users log in using an existing account (like Google) instead of creating a new username and password.
We use Google OAuth so that Algoleap employees can simply click 'Sign in with Google' and use their existing @algoleap.com Google account.
We configure it to ONLY allow @algoleap.com email addresses — anyone trying to log in with a personal Gmail account will be blocked.
You do not build the login system from scratch. Google handles the actual authentication. 
Supabase
 connects to Google on your behalf.
What you will use it for: The login page. One button, secure login, zero password management.
2.5 
Vercel
 — Hosting the Website
CONCEPT  What This Means
Hosting means making your website accessible on the internet. When you build a website on your laptop, only you can see it. 
Vercel
 puts it on the internet so everyone at Algoleap can access it.
Vercel
 is the company behind Next.js, so the two work perfectly together.
Every time you push code to GitHub (a code storage platform), 
Vercel
 automatically rebuilds and republishes your website. This means deploying a change is as simple as saving your code.
Vercel
 has a generous free tier that is more than sufficient for 50 artifacts and a few dozen users.
What you will use it for: Making the portal available at a real URL (e.g., hub.algoleap.com) that every employee can open in their browser.
2.6 GitHub — Saving and Sharing Your Code
CONCEPT  What This Means
GitHub is a platform for storing code online, tracking every change you make, and collaborating with others.
Every time you save a batch of changes, you make a 'commit'. Every commit is saved permanently — you can always go back to a previous version if something breaks.
Vercel
 watches your GitHub repository and automatically deploys every time you push new code to the 'main' branch.
What you will use it for: Storing all your code online, and triggering automatic deployments to 
Vercel
.
2.7 React Hook Form + Zod — Building Safe Admin Forms
CONCEPT  What This Means
React Hook Form is a library that makes it easy to build forms in React — things like the 'Add Artifact' form in the admin panel.
Zod is a library that validates data. For example, it makes sure the OneDrive URL field actually contains a URL, and that required fields are not left empty.
Together they ensure that the knowledge management owner cannot accidentally submit an artifact with missing or incorrect data.
What you will use it for: The artifact intake form and the edit form in the admin panel.
2.8 
Lucide
 React — Icons
CONCEPT  What This Means
Lucide
 React is a library of clean, consistent icons (small SVG images used as visual indicators in the UI — search icon, filter icon, external link icon, etc.).
We import only the icons we need, keeping the app fast.
What you will use it for: All icons throughout the portal — search, filter, open link, settings, user avatar, etc.

Section 5: Setting Up the Database in 
Supabase
A database stores structured data in tables — similar to spreadsheet tables with rows and columns. We need two tables: one for artifacts (the main 
catalog
) and one for user profiles (to track who is an admin).
CONCEPT  What This Means
SQL (Structured Query Language) is the language used to create and manage database tables. You do not need to deeply understand it yet — just run the commands provided.
In 
Supabase
, you run SQL commands in the 'SQL Editor' — it is like a text editor for your database.
A 'table' is like a spreadsheet tab. Each 'row' is one record (one artifact). Each 'column' is one field (title, type, URL, etc.).
5.1 Open the 
Supabase

Log in to supabase.com and open your project
In the left sidebar, click 'SQL Editor'

This is the main table that stores all artifact records. Copy the entire block below, paste it into the SQL Editor, and click 'Run'.

  id             UUID DEFAULT

() PRIMARY KEY,
  title          TEXT NOT NULL,
  description    TEXT,

  TEXT NOT NULL,

   TEXT[],

5.3 Create the User Profiles Table
This table tracks each user's role (viewer or admin). Run this in a new query:
CREATE TABLE profiles (
  id     UUID REFERENCES 
auth.users
(id) ON DELETE CASCADE PRIMARY KEY,
  email  TEXT,
  role   TEXT DEFAULT 'viewer'
);
CONCEPT  What This Means
REFERENCES 
auth.users
(id) links this table to 
Supabase's
 built-in users table — every profile belongs to a real user account.
ON DELETE CASCADE means if a user is deleted, their profile is also automatically deleted.
The 'role' field has only two meaningful values: 'viewer' (regular employee) and 'admin' (knowledge management owner).
5.4 Create the Auto-Profile Trigger
This trigger automatically creates a profile row every time a new user signs in for the first time. Run this as a new query:
CREATE OR REPLACE FUNCTION 
public.handle_new_user
()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO 
public.profiles
 (id, email)
  VALUES (NEW.id, 
NEW.email
);
  RETURN NEW;
END;
$$ LANGUAGE 
plpgsql
 SECURITY DEFINER;
CREATE TRIGGER 
on_auth_user_created
  AFTER INSERT ON 
auth.users
  FOR EACH ROW EXECUTE PROCEDURE 
public.handle_new_user
();
CONCEPT  What This Means
A trigger is a database function that runs automatically when something happens.
This trigger listens for new user sign-ins and automatically adds them to the profiles table with role = 'viewer'.
You will then manually update the KM owner's role to 'admin' in the Table Editor.
5.5 Set Up Row Level Security (RLS) Policies
Row Level Security controls who can read or modify which rows. This is what keeps the portal secure. Run these as a single query:
-- Enable RLS on both tables
ALTER TABLE artifacts ENABLE ROW LEVEL SECURITY;
ALTER TABLE profiles  ENABLE ROW LEVEL SECURITY;
-- All logged-in users can READ artifacts
CREATE POLICY 
artifacts_read
 ON artifacts
  FOR SELECT TO authenticated USING (true);
-- Only admins can INSERT, UPDATE, DELETE artifacts
CREATE POLICY 
artifacts_write
 ON artifacts
  FOR ALL TO authenticated
  USING (EXISTS (
    SELECT 1 FROM profiles
    WHERE id = 
auth.uid
() AND role = 'admin'
  ));
-- Users can only read their own profile
CREATE POLICY 
profiles_read
 ON profiles
  FOR SELECT TO authenticated USING (id = 
auth.uid
());
CONCEPT  What This Means
RLS ensures that even if someone gets a database key, they can only do what their role allows.
auth.uid
() is a 
Supabase
 function that returns the ID of the currently logged-in user.
The admin write policy checks: 'Does the current user have role = admin in the profiles table?' If yes, allow the action.
CHECK  Expected Result
All four SQL blocks run without errors. RLS is now enabled — only authenticated users can see artifacts, and only admins can modify them.
